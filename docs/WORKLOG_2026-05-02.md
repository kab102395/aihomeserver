## Phase 1: Coder Executor Pipeline

### Root problem it solves
The Snake game canonical failure: model produced only `Cargo.toml`, Repair kept retrying
the wrong step, the ZIP was never created, and the critic had no contract to enforce.
All failures were invisible because verification was entirely LLM-opinion based.

|------|---------|
| `src/coder/intent.rs` | `detect_coding_intent()` — keyword heuristic, no LLM call |
| `src/coder/adapter.rs` | `LanguageAdapter` trait + `AdapterManifest` struct |

### Modified files

**`src/tools/filesystem.rs`** — Added `zip_dir` / `zip` action using the `zip = "2"` crate.
Walks source dir, applies exclude patterns (default: `target/`, `.git/`), writes deflated
zip, returns `{ path, entries, entry_count, size_bytes }`.

**`src/state.rs`** — Added `coding_intent: Option<CodingIntent>` and
`execution_manifest: Option<ExecutionManifest>` to `SystemState`. Added `ProjectCard`
SSE event variant. Added `CodingClassifier` to `OrchestratorNode`.

**`src/orchestrator.rs`** — Inserted `CodingClassifier` between Intake and Planner.
After every `ToolExecution`, if `coding_intent` is set and `execution_manifest` is
loaded, runs `coder::verify()`, emits `ProjectCard` SSE, and stores
`artifact_verification` artifact. Added `try_load_execution_manifest()`,
`collect_build_exit_codes()`, `count_files_written()` helpers.

**`src/nodes/planner.rs`** — Injects CODING PLAN CONTRACT when `coding_intent` is set:
mandatory manifest step → file writes → verification → zip_dir → answer. Lists
supported filesystem actions explicitly (prevents hallucinated `create_file` calls).

**`src/nodes/executor.rs`** — Coding-aware system prompt for tool steps (appends
adapter `system_prompt_addition`). `coding_manifest_prompt()` used for the manifest
generation step, giving the model the exact `ExecutionManifest` JSON schema.

**`src/nodes/critic.rs`** — Deterministic manifest check before LLM critic. If
`artifact_verification` shows missing files, missing artifacts, or build failure,
fails immediately with confidence=1.0 — no LLM tokens spent.

**`src/nodes/repair.rs`** — `CodingRepairTarget` enum. Reads `execution_manifest` +
`artifact_verification` and injects a targeted directive into the repair prompt:
`WriteFile` / `PatchSource` / `RunPackage` instead of a generic "fix it".

**`src/api/server.rs`** — Added `ProjectCard` to the SSE event type match.

**`src/api/ui.rs`** — `project_card` SSE handler renders a status card in the
reasoning panel: language badge, project name, status (building/verified/partial/failed),
file count, build badge (✓/✗), Download ZIP link. CSS included.

**`Cargo.toml`** — Added `zip = "2"` dependency.

## Bug Fixes (post-Phase-1 test session)

### Bug 1 — `<think>` tag leak into answers
**File:** `src/llm/ollama.rs`

qwen3 sometimes embeds `<think>...</think>` blocks directly in `message.content` even
when the separate `message.thinking` field is being used. `strip_think_tags()` is now
applied to every `chat()` and `chat_stream()` return value as a defensive layer.

### Bug 2 — Research answers hallucinating specifics
**File:** `src/nodes/executor.rs`

Added `GROUNDING_GUARDRAIL` — injected at the END of the user message for every
`requires_facts: true` step. Instructs the model to write `[not in research data]`
for any specific claim not present in artifacts, and refuse to answer if artifacts
are empty. Placed at end of context to maximise attention weight. Also strengthened
`SYSTEM_PROMPT_LLM` to explicitly forbid stating version numbers or dates from memory.

### Bug 3 — Critic auto-passing hallucinated research answers
**File:** `src/nodes/critic.rs`

Added `RESEARCH_CRITIC_PROMPT` — a dedicated hallucination detector that runs for any
step with `requires_facts=true` + `facts` artifact present, bypassing the low-risk
auto-pass. Checks whether specific version numbers, dates, or stats in the answer
are actually present in the facts artifact. Correctly caught examples like:
- `"Answer mentions 'breeding' in Rust 2026 roadmap — not in facts (conflating game Rust)"`
- `"Answer claims no-GIL introduced in 3.13 but facts show it's in 3.14"`

### Bug 4 — Terrible fallback search queries
**File:** `src/grounding.rs`

`decompose_search_queries()` was generating two useless copies of the raw user request
for non-game topics. Added `extract_tech_subject()` which recognises ~40 languages and
tools. For "latest Rust versions" it now generates:
- `"Rust latest release what's new"`
- `"Rust recent versions changelog"`
- `"Rust release notes blog 2025 2026"`
- `"Rust new features updates site:reddit.com"`

### Bug 5 — Critic firing too early on multi-step coding plans
**File:** `src/nodes/critic.rs`

`FAST_CRITIC_PROMPT` was checking overall plan completion criteria ("zip must exist")
after EVERY step including step 1 (search). Caused Go/Snake coding tasks to enter
infinite repair loops before the zip step ever ran. New prompt is explicit:
"Evaluate THIS ONE STEP only — later steps have not run yet."

### Bug 6 — Critic model producing garbage on large contexts
**File:** `src/nodes/critic.rs`

When full artifact context (including raw HTML fetch results) was fed to the critic,
it produced corrupted output like `.waitKey` and `-pdf`. Added `truncate_artifacts()`
and `truncate_json()` helpers: fetch/search results capped at 200-char snippets, total
artifact context capped at ~4K chars. Critic failures now auto-pass with a log warning
instead of triggering a broken repair loop.

### Bug 7 — Coding tasks answered from KB instead of creating files
**File:** `src/nodes/planner.rs`

The planner found a prior Snake KB entry and returned "Task completed." instead of
creating files. Added explicit `"Do NOT answer from knowledge base"` instruction to
the coding task context injection. Coding tasks always need real file creation.

### Bug 8 — Answer preamble reasoning leaking
**File:** `src/nodes/executor.rs`

`SYSTEM_PROMPT_LLM` now explicitly instructs: do not start with "Okay", "Let me",
"First I need to", "Looking at", or any other reasoning preamble. Start immediately
with the answer content.

### Bug 9 — Hardcoded years in search queries
**File:** `src/grounding.rs`

Query generator had hardcoded `"2024 2025"` strings. Replaced with
`chrono::Local::now()` so queries always include the actual current and prior year.
Will self-update on January 1st without any code change.

## What Still Needs Work

### High priority

**Repair loop re-searching for coding tasks** — When repair fires for a zip step
failure, the model regenerates `parallel_search` instead of the corrected `zip_dir`
tool call. The `CodingRepairTarget::RunPackage` directive is injected but the model
ignores it. May need to deterministically generate the zip_dir tool call JSON in
repair rather than asking the LLM to do it.

**Model keeps hallucinating through repair cycles** — Even after the research critic
correctly catches a hallucination and triggers repair, the model invents different
wrong specifics in the repaired answer. The repair prompt needs a stronger "if you
don't have this fact in the artifacts, write exactly `[not in research data]` and
move on" enforcement. Consider: if repair cycle >= 2 on a research task, inject the
facts artifact verbatim into the repair prompt and force the model to quote directly
from it rather than synthesising.

**Coding task execution reliability** — Snake produced "Task completed." on first
run (KB bypass was the cause — now fixed), but coding tasks need more live testing
to confirm the execution manifest step, file-by-file writes, verification, and
`zip_dir` all fire correctly in sequence. The `execution_manifest` must be produced
by the planner for `artifact_verification` to work in the critic.

### Medium priority

**`zip_dir` result not surfacing in critic** — The FAST_CRITIC_PROMPT fix means the
critic no longer fires early, but we need to confirm the critic correctly validates
the `zip_dir` result artifact when it IS the current step. The `zip_dir` action
returns `{ path, entries, entry_count, size_bytes, success }` — the critic should
check `success=true` and `size_bytes > 0`.

**Critic JSON parse failures under load** — Fixed the most common case (large context
→ corrupt output) but the critic model occasionally produces invalid JSON for other
reasons. Consider: auto-retry `complete_json` once on parse failure before accepting
the fallback auto-pass.

**Research quality for non-grounded answers** — For questions where `request_needs_grounding()`
doesn't fire (no "version", "latest", "patch" keywords), the model can still hallucinate.
The grounding heuristic could be broadened — e.g., any question about "current state of X"
or "what does X support" should probably trigger grounding.

### Low priority / Phase 2 (from spec)

- Java, C#, C++, Lua language adapters
- Full `ToolchainDetector` node (currently handled via shell commands in plan)
- Profile resolver UI dropdowns (let user pick `cli` vs `terminal-game-crossterm` etc.)
- Library intelligence layer (grounded docs per-library to prevent invented API names)
- Dependency policy enforcement (block `rand = "*"` type version specs)
- Eval suite: Rust Snake, Go HTTP server, Python script — repeatable end-to-end tests
- Search query decomposition for non-tech non-game topics (currently falls back to
  cleaned raw request, which is still not great)
